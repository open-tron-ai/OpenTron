## Architecture Decisions

### 1. CLI Delegation Pattern

**Rationale**: Python OpenTron has a complex, mature CLI. Rather than rewriting all CLI commands immediately, we forward to the Python CLI, preserving functionality while Java porting progresses.

**Implementation**:
```java
public static int runPythonCli(String... args) throws IOException, InterruptedException {
    String py = getPythonExecutable();
    List<String> cmd = new ArrayList<>();
    cmd.add(py);
    cmd.add("-m");
    cmd.add("OpenTron.cli");
    if (args != null) {
        for (String a : args) cmd.add(a);
    }
    ProcessBuilder pb = new ProcessBuilder(cmd);
    pb.inheritIO();
    Process p = pb.start();
    return p.waitFor();
}
```

**Advantages**:
- Full CLI functionality available immediately
- Gradual conversion strategy possible
- CLI testing remains on Python side until migration

**Disadvantages**:
- Requires Python runtime
- Process spawning overhead
- Cannot easily inspect or modify CLI behavior from Java

### 2. Java Package Mirroring

**Python Structure**:
```
src/OpenTron/
├── core/
├── cli/
├── agents/
├── tools/
└── ...
```

**Java Structure** (exact mirror):
```
src/main/java/io/OpenTron/
├── core/
├── cli/
├── agents/
├── tools/
└── ...
```

**Rationale**: Predictable navigation; module `OpenTron.X.Y` → `io.OpenTron.X.Y`

### 3. Type System Conversion

**Python → Java**:
- `Union[Type1, Type2]` → Superclass or marker interface
- `Optional[T]` → `T` with null checks or `Optional<T>`
- `Dict[K, V]` → `Map<K, V>`
- `List[T]` → `List<T>`
- `Enum` → Java `enum`
- `@dataclass` → POJO with getters/setters
- `@register` decorator → Imperative `Registry.register(key, value)`

**Example (Role enum)**:
```python
# Python
class Role(str, Enum):
    SYSTEM = "system"
    USER = "user"
    ASSISTANT = "assistant"
    TOOL = "tool"
```

```java
// Java
public enum Role {
    SYSTEM("system"),
    USER("user"),
    ASSISTANT("assistant"),
    TOOL("tool");
    
    private final String value;
    
    Role(String value) {
        this.value = value;
    }
    
    public String getValue() { return value; }
}
```

### 4. Registry Pattern Conversion

**Python** (decorator-based):
```python
@ToolRegistry.register("my_tool")
class MyTool:
    pass
```

**Java** (imperative):
```java
// In static initializer or factory
ToolRegistry.registerValue("my_tool", new MyTool());
```

**Benefits**:
- Explicit registration order control
- No reflection magic required
- Static type checking

## Known Limitations & Workarounds

### 1. Dynamic Imports
**Problem**: Python code uses `importlib.import_module()` to dynamically load modules based on config.

**Python Example**:
```python
engine_module = importlib.import_module(f"OpenTron.engine.{engine_key}")
```

**Java Limitation**: Static class loading at compile time.

**Workaround**: Factory pattern with switch statement or `ServiceLoader` SPI.

**Action Required**: Manual conversion of dynamic imports to static factories.

---

### 2. Async/Await
**Problem**: Python uses `asyncio` heavily; Java has different concurrency model.

**Python Example**:
```python
async def run(query):
    result = await agent.run(query)
    return result
```

**Java Options**:
1. `Thread` and `join()`
2. `ExecutorService` and `Future`
3. `CompletableFuture` (preferred for callbacks)
4. `virtual threads` (Java 19+)

**Status**: Pending; currently all CLI code delegates to Python.

---

### 3. Pydantic Models
**Problem**: Python uses Pydantic for validation and serialization; Java requires manual mapping.

**Python Example**:
```python
class Message(BaseModel):
    role: Role
    content: str
    tool_calls: Optional[List[ToolCall]] = None
    
    class Config:
        use_enum_values = True
```

**Java Solution** (simplified):
```java
public class Message {
    private Role role;
    private String content;
    private List<ToolCall> toolCalls;
    
    // Getters/setters
    public Role getRole() { return role; }
    public void setRole(Role role) { this.role = role; }
    // ...
}
```

**Loss**: Runtime validation; serialization must be handled manually (Jackson, Gson, or custom).

---

### 4. Decorators (Beyond @register)
**Problem**: Python uses decorators for caching, validation, logging, etc.

**Python Example**:
```python
@cache
@validate_input
def compute(x):
    return x * 2
```

**Java Solution**:
1. Direct implementation (no decorator)
2. Wrapper methods
3. Decorator pattern (class-based)
4. AOP (Aspect-Oriented Programming) via AspectJ

**Status**: Most decorators simplified to direct implementation.

---

### 5. Context Managers
**Problem**: Python's `with` statement and context managers (`__enter__`, `__exit__`).

**Python Example**:
```python
with open(file) as f:
    data = f.read()
```

**Java Solution**: Try-with-resources (Java 7+).
```java
try (var f = new FileReader(file)) {
    String data = IOUtils.toString(f);
}
```

**Status**: Straightforward conversion; implemented where needed.

---

### 6. Rust Bridge
**Problem**: `src/OpenTron/_rust_bridge.py` is a wrapper around native Rust code.

**Status**: ⏳ Pending investigation.

**Options**:
1. JNI bindings to existing Rust library
2. Re-implement in Java
3. Spawn Rust binary via `ProcessBuilder`

**Recommendation**: Defer until core Java conversion complete.

---

### 7. Process Environment Mutation
**Problem**: Python code does `os.environ["KEY"] = value` and expects child processes to see it.

**Java Limitation**: `System.setProperty()` affects only the JVM; child processes don't inherit.

**Python Code**:
```python
os.environ["OPENAI_API_KEY"] = key
subprocess.run(["python", "-m", "OpenTron"])  # Sees OPENAI_API_KEY
```

**Java Limitation**:
```java
System.setProperty("OPENAI_API_KEY", key);
ProcessBuilder pb = new ProcessBuilder("java", "...");
// pb does NOT see the property; must pass via pb.environment()
```

**Workaround**:
```java
pb.environment().put("OPENAI_API_KEY", key);
```

**Status**: Documented; affected code will need review.
